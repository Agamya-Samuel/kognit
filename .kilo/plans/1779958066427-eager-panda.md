# Auth System Analysis & Improvements

## 1. Current Architecture (apps/api)

### Auth module layout
- `apps/api/src/modules/auth/auth.controller.ts` — endpoints: `/auth/google[/callback]`, `/auth/register/{request,verify,complete}`, `/auth/login`, `/auth/logout`, `/auth/refresh`, `/auth/me`, `/auth/forgot-password`, `/auth/reset-password`, `/auth/change-password`, `/auth/email-verification/{request,verify}`.
- `apps/api/src/modules/auth/auth.service.ts` — business logic for the email-first registration flow, login, OAuth, refresh-token rotation, password reset.
- `apps/api/src/modules/auth/guards/roles.guard.ts` — single-role check via `@Roles()` metadata (`requiredRoles.includes(user.role)`).
- `apps/api/src/modules/auth/strategies/google-oauth.strategy.ts` — Google passport strategy that calls `authService.handleOAuthLogin`.
- `apps/api/src/db/schema/users.ts` — `role` is a single-value enum column with default `'student'`.
- `apps/api/src/db/schema/enums.ts` — `userRoles` enum: `['student', 'instructor', 'admin', 'institution_admin']`.

### Frontend portals (today)
- `apps/web-student/src/app/auth/register/page.tsx` — calls `/auth/register/{request,verify,complete}`.
- `apps/web-instructor/src/app/auth/register/page.tsx` — same flow, hits the same endpoints (no role/platform signal).
- `apps/web-admin/src/app/auth/register/page.tsx` — same flow, hits the same endpoints.
- `packages/api-client/src/services/auth.ts` — shared client used by all three apps; sends only `{ email, code, name, password }`.

### Role assignment today
- Email/password registration (`auth.service.ts:158`) creates the user with **`role: 'student'`** — correct.
- OAuth (Google) registration (`auth.service.ts:485-494`) creates the user with **`role: 'instructor'`** — **this is the bug**.
- DB default role (`users.ts:9`) is `'student'`.
- `RolesGuard` only checks `user.role` (single role); registration endpoints are `@Public()` so no role check applies on signup.

### RBAC observations
- Single-role model is already in place; no `user_roles` join table.
- No portal/origin awareness — the API has no idea whether a request came from web-student, web-instructor, or web-admin.
- No "pending/approval" state on `users` — `isActive` and `isVerified` are the only lifecycle flags. The existing `approvalStatus` enum is unused on `users`.

## 2. Issues Identified

### Issue 1 — Wrong role on OAuth signup
`auth.service.ts:489` hardcodes `role: 'instructor'` for new OAuth users. Any user signing in with Google for the first time is incorrectly marked as `instructor`, regardless of the originating portal. The email/password flow does not have this bug (it always sets `'student'`), but it also has the opposite problem: it always creates students even when the request came from web-instructor or web-admin.

### Issue 2 — Registration is not restricted per portal
- All three Next.js apps ship a `/auth/register` page.
- The API has a single public registration endpoint set; it does not know or enforce which portal initiated the call.
- Anyone can self-register an "instructor" or "admin" account via the wrong portal's UI, and the backend would happily create a `student` (email path) or `instructor` (OAuth path) account.

### Issue 3 — No multi-portal access policy
- A `student`-role user can call `/auth/login` from web-instructor or web-admin successfully because login does not check role-vs-portal.
- Conversely, there is no clean upgrade path (student → instructor) and no admin-invite flow.

---

## 3. Decisions (confirmed with user)

- **Account model:** Single account, single role per user. Each portal login must enforce that the user's role matches that portal.
- **Instructor onboarding:** Keep the registration UI on web-instructor, but new instructor signups are created in a `pending` state and must be approved by an admin before they can log in. (This requires a new `approvalStatus` field on `users`.)
- **Admin onboarding:** No public registration. Admin accounts are created by seed or by an existing admin/super-admin only. The `/auth/register` page is removed from web-admin.

## 4. Proposed Changes

### 4.1 Schema changes (`apps/api/src/db/schema/users.ts`)
- Add `approval_status` column to `users` reusing the existing `approvalStatus` enum (`['pending', 'approved', 'rejected']`), default `'approved'`.
  - Existing rows backfill to `'approved'` via migration.
  - New `student` accounts → `'approved'` immediately.
  - New `instructor` accounts → `'pending'` until an admin approves.
- New Drizzle migration generated via `pnpm --filter @edutech/api db:generate` (do not hand-edit SQL).

### 4.2 Registration endpoints — make portal-aware
Add an explicit, server-validated portal/intent parameter rather than trusting headers:

- Update DTOs in `apps/api/src/modules/auth/dto/auth.dto.ts`:
  - `RequestEmailVerificationDto` and `CompleteRegistrationDto` get a new field `intent: 'student' | 'instructor'` (zod/class-validator enum, required).
  - Keep payload shape backward-compatible by defaulting missing `intent` to `'student'` for one release with a deprecation log.
- `auth.service.ts`:
  - `requestRegistrationVerification(email, intent)` — store `intent` alongside the verification code in Redis so it cannot be tampered with between steps.
  - `completeRegistration(email, code, name, password)` — read `intent` from the cached pending registration; map to role + approval status:
    - `student` → `role: 'student'`, `approvalStatus: 'approved'`, login allowed.
    - `instructor` → `role: 'instructor'`, `approvalStatus: 'pending'`, login blocked with a clear "pending approval" error.
  - Reject `intent === 'admin'` (or any other value) at the DTO level — admin signup is not a public flow.
- `apps/api/src/modules/auth/auth.controller.ts` — pass `intent` through; document in Swagger.
- `packages/api-client/src/services/auth.ts` — extend `requestRegistrationVerification` and `completeRegistration` to accept and forward `intent`.

### 4.3 Fix OAuth role assignment
`auth.service.ts:485-494` (`handleOAuthLogin`):
- Change hardcoded `role: 'instructor'` to `role: 'student'` and `approvalStatus: 'approved'` for the default Google sign-up case.
- Accept an optional `intent` argument propagated from the OAuth `state` parameter (already used for redirect URLs). The Google strategy will read `state.intent` (we will move to a structured/base64-JSON `state` instead of plain redirect string).
- For `intent === 'instructor'` via OAuth, create with `role: 'instructor'` + `approvalStatus: 'pending'` and surface a "pending approval" page on the frontend.
- For unknown/missing intent → default `'student'`.
- Update `apps/api/src/modules/auth/strategies/google-oauth.strategy.ts` to parse the structured state and forward `intent`.

### 4.4 Portal-vs-role enforcement at login
Add an optional `portal` parameter to `/auth/login` (and OAuth callback target) that the frontend always sets. On successful credential check:
- web-student portal → require `role === 'student'`.
- web-instructor portal → require `role === 'instructor'` AND `approvalStatus === 'approved'`.
- web-admin portal → require `role === 'admin'` ONLY. **`institution_admin` must NOT be granted access to web-admin** — they are a separate role with different permissions and should have their own portal and endpoints.
- Mismatch returns `403` with code `WRONG_PORTAL` and a message guiding the user to the correct portal.
- Keep `/auth/me` as the source of truth for role; UIs use it to redirect on mismatch.

Implementation:
- Extend `LoginDto` with `portal?: 'student' | 'instructor' | 'admin'`.
- Add a private helper `assertPortalAccess(user, portal)` in `auth.service.ts` called from `login` and `handleOAuthLogin`.
- Pending instructors get a distinct error code `ACCOUNT_PENDING_APPROVAL` (no token issued).

### 4.5 Frontend changes
- `apps/web-admin/src/app/auth/register/page.tsx` — **delete** the page, remove the "Sign up" link from `apps/web-admin/src/app/auth/login/page.tsx` (`onSignUpClick`). Replace with a static notice: "Admin accounts are provisioned by your organization."
- `apps/web-instructor/src/app/auth/register/page.tsx` — keep the page, but:
  - Pass `intent: 'instructor'` in both `requestRegistrationVerification` and `completeRegistration` calls.
  - After successful completion, route to a new `/auth/pending` page explaining that an admin must approve the account; do not auto-login (the API will not return tokens for pending accounts).
- `apps/web-student/src/app/auth/register/page.tsx` — pass `intent: 'student'` explicitly.
- All three login pages — pass the appropriate `portal` value, and on `WRONG_PORTAL` show "This account is for the X portal" with a link.

### 4.6 Admin approval surface — EXTEND existing endpoints
- **EXTEND** `GET /admin/users` (already exists in admin.controller.ts) — add `?approvalStatus=pending` filter param. The existing `UsersRepository.findMany()` already supports role filter; add approvalStatus filter alongside it.
- **REUSE** `PATCH /admin/instructors/:id/approve` (already exists) — the existing `AdminService.approveInstructor()` already sets `approvalStatus='approved'` and promotes role to `instructor`. No changes needed.
- **REUSE** `PATCH /admin/instructors/:id/reject` (already exists) — same pattern for rejection with reason.
- Audit log entry on each transition (use existing `audit-logs` schema).
- Add a "Pending instructors" section in `apps/web-admin` dashboard (out-of-scope for this auth fix but flagged).

### 4.7 Admin/super-admin bootstrap
- Document and verify `apps/api/src/db/seed.ts` creates at least one super-admin.
- Add a CLI/seed-only path for additional admin creation; never expose via HTTP.

## 6. Proposed Flows

### Student signup (web-student)
1. User → `/auth/register` (web-student).
2. Frontend → `POST /auth/register/request` `{ email, intent: 'student' }`.
3. API stores pending registration with `intent='student'` in Redis, emails 6-digit code.
4. User → `POST /auth/register/verify` → `POST /auth/register/complete`.
5. API creates user `role='student'`, `approvalStatus='approved'`, `isVerified=true`, returns tokens.
6. User logged in to student portal.

### Instructor signup (web-instructor)
1. User → `/auth/register` (web-instructor).
2. Frontend → `POST /auth/register/request` `{ email, intent: 'instructor' }`.
3. Same verify step.
4. `POST /auth/register/complete` creates user `role='instructor'`, `approvalStatus='pending'`, **no tokens issued**.
5. Frontend redirects to `/auth/pending`.
6. Admin approves in web-admin → `approvalStatus='approved'`, notification email sent.
7. User logs in via web-instructor; backend asserts `role==='instructor' && approvalStatus==='approved' && portal==='instructor'`.

### Admin login (web-admin)
1. No registration UI.
2. Admin uses `/auth/login` with `portal: 'admin'`.
3. Backend asserts `role === 'admin'` ONLY. `institution_admin`, `student`, `instructor` all get `403 WRONG_PORTAL`.

### Cross-portal attempt
- Student account tries to log in on web-instructor → `403 WRONG_PORTAL` with helpful message + link to student portal.

### Google OAuth
- `state` becomes structured: `{ redirect, intent }` (base64 JSON).
- Same role/approval mapping as email path, including `pending` for instructor intent.
- Default intent when missing → `student`.

## 9. NEW — Student Onboarding Flows (Extended)

The user has two student onboarding flows that need to be implemented. Both flows capture extended profile data after the initial account creation step.

### 9.1 Flow A — Website Self-Signup (Email/Password or OAuth)

**Current state:** The email-first registration flow (`/auth/register/request → /verify → /complete`) and OAuth create a user with `role: 'student'` and only capture `email`, `name`, and `password` (OAuth: no password). No extended profile data is captured.

**Required:** After the initial account is created and email is verified (via code or OAuth success), the student must complete an additional onboarding step to capture:

| Field | Source | Editable? | Notes |
|---|---|---|---|
| `name` | Pre-filled from registration | Yes, **during onboarding only** | After onboarding completes, name is locked — requires admin to change |
| `mobile` | User input | Yes | Phone number with validation |
| `address` | User input | Yes | Full street address |
| `city` | User input | Yes | |
| `state` | User input | Yes | |
| `pinCode` / `zipCode` | User input | Yes | Alphanumeric, max 10 chars |
| `country` | User input | Yes | ISO country code or free text |
| `affiliatedInstituteId` | Read-only | No | FK to `institution_accounts.id`. Set by admin during bulk import (Flow B). For self-signup students, defaults to `NULL` (displayed as "NA" in UI). |

**Proposed implementation (reusing existing endpoints):**

1. **Extend `GET /auth/me`** (existing auth.controller.ts):
   - Add `onboardingCompleted` and `studentProfile` fields to the response.
   - Frontend calls this on app load to decide whether to redirect to `/onboarding` or `/dashboard`.

2. **Extend `PATCH /users/profile`** (existing users.controller.ts):
   - Extend `UpdateUserProfileDto` with student profile fields: `mobile`, `address`, `city`, `state`, `pinCode`, `country`, `affiliatedInstituteId`.
   - Extend `UsersService.updateProfile()` to:
     - Create or update the `student_profiles` record in a transaction.
     - Set `onboardingCompleted: true` on the `users` table.
     - Reject `name` changes if `onboardingCompleted` is already `true` (name locking).
   - **This eliminates the need for a separate `POST /auth/student-onboarding/complete` endpoint.**

3. **Add minimal `GET /users/onboarding-status`** (new, but lightweight):
   - Returns `{ onboardingCompleted: boolean, affiliatedInstituteId: number | null, affiliatedInstituteName: string | null }`.
   - This is lighter than calling `GET /auth/me` and can be called from the onboarding page for the initial load.
   - **Alternative:** Just use `GET /auth/me` and skip this endpoint entirely. The extra data is negligible. **Decision: skip this endpoint, use `GET /auth/me` only.**

4. **Frontend flow (web-student):**
   - After email/password registration completes (tokens received) → redirect to `/onboarding`.
   - After OAuth login (if `isNewUser === true`) → redirect to `/onboarding`.
   - `/onboarding` page calls `GET /auth/me`, pre-fills name, and shows the form.
   - On submit → calls `PATCH /users/profile` with all onboarding fields, then redirects to dashboard.
   - Authenticated layout (`(authenticated)/layout.tsx`) checks `onboardingCompleted` on `GET /auth/me` and redirects to `/onboarding` if false (prevents dashboard access before onboarding).
   - After onboarding is complete, name becomes non-editable in profile settings (enforced by UI + API validation in `PATCH /users/profile`).

### 9.2 Flow B — Bulk Onboard by Admin (CSV Import)

**Process:**

1. **Admin creates an institution account** in `web-admin` (existing `institution_accounts` table — already exists).
2. **Admin uploads a CSV file** containing student records. Expected columns: `name`, `email` (institution is implied by the selected institution account).
3. **API processes the CSV:**
   - Validates each row (email format, uniqueness, name non-empty).
   - Creates a user record for each valid student with:
     - `role: 'student'`
     - `name: <from CSV>`
     - `email: <from CSV>`
     - `isVerified: false`
     - `isActive: true`
     - `onboardingCompleted: false`
   - Creates a `student_profiles` record linking the student to the institution (`affiliatedInstituteId`).
    - Generates a **secure activation token** with **1200-day expiry**.
    - Stores the activation token in the extended `email_verifications` table with `purpose: 'student_activation'`.
   - Sends an activation email to each student with a unique link.

4. **Activation email link:**
   - URL format: `https://web-student.example.com/auth/activate?token=<activation_token>`
   - Clicking the link opens the activation page.

5. **Activation page (web-student `/auth/activate`):**
   - Validates the token (existence, expiry, unused).
   - **Step 1:** Set a strong password (with same validation as registration).
   - **Step 2:** Complete profile form — pre-fills name (editable during this step), then asks for mobile, address, city, state, PIN/ZIP, country. Affiliated institute is shown as read-only.
   - On submit → marks token as used, sets `isVerified: true`, `onboardingCompleted: true`, and logs the user in (returns tokens).

**Proposed implementation:**

   - Validates each row (email format, uniqueness, name non-empty).
1. **Extend `email_verifications` table** (replaces the planned `student_activation_tokens` table):
   - Add `purpose` enum column with values `'email_verify'` and `'student_activation'`.
   - Increase `tokenHash` length from 60 to 255 to support long activation tokens.
   - Change `onDelete` from `'restrict'` to `'cascade'` so tokens are cleaned up when a user is deleted.
   - Existing `EmailVerificationService` uses `purpose: 'email_verify'` (backfill migration sets this).
   - New `StudentActivationService` uses `purpose: 'student_activation'`.
   - Both services share the same table and repository, but are isolated by `purpose`.

   ```typescript
   export const emailVerifications = pgTable('email_verifications', {
     id: serial('id').primaryKey(),
     userId: integer('user_id').notNull().references(() => users.id, { onDelete: 'cascade' }),
     tokenHash: varchar('token_hash', { length: 255 }).notNull(),
     purpose: emailVerificationPurpose('purpose').notNull().default('email_verify'),
     expiresAt: timestamp('expires_at').notNull(),
     verified: boolean('verified').notNull().default(false),
     createdAt: timestamp('created_at').notNull().defaultNow(),
   }, (table) => [
     index('email_verifications_token_hash_idx').on(table.tokenHash),
     index('email_verifications_user_id_idx').on(table.userId),
     index('email_verifications_purpose_idx').on(table.purpose),
   ]);
   ```

2. **New endpoint `POST /admin/institutions/:id/students/import`** (admin-only, multipart/form-data):
   - Accepts CSV file upload.
   - Processes each row, creates users + student_profiles + activation tokens.
   - Sends activation emails.
   - Returns `{ successCount, failureCount, errors: [{ row, email, reason }] }`.

3. **New endpoint `POST /auth/student-activation/validate`** (public):
   - Accepts `{ token }`.
   - Validates token (exists, not expired, not used).
   - Returns `{ valid: true, email: string, name: string, institutionName: string | null }` (no sensitive data).

4. **New endpoint `POST /auth/student-activation/complete`** (public):
   - Accepts `{ token, password, name, mobile, address, city, state, pinCode, country }`.
   - Validates token, password strength, and profile fields.
   - Updates user: `passwordHash`, `name` (if edited), `isVerified: true`, `onboardingCompleted: true`.
   - Updates `student_profiles` with profile data.
   - Marks activation token as `used: true`.
   - Returns `{ user, tokens }` (auto-login after activation).

5. **CSV parsing:**
   - Use a lightweight CSV parser (e.g., `csv-parser` or `papaparse` on the server).
   - Validate email uniqueness against existing users.
   - Handle duplicates gracefully (skip + report in errors).

- Add validation in the existing `PATCH /users/profile` (`UsersService.updateProfile()`) that rejects `name` changes if `onboardingCompleted === true` and `role === 'student'`. Return `403` with code `NAME_LOCKED` and message "Name cannot be changed after onboarding. Contact an administrator."

### 9.5 Token Storage — Reusing `email_verifications`

Instead of creating a new `student_activation_tokens` table, the existing `email_verifications` table is extended:

**Changes to `email_verifications` schema:**
- Add `purpose` enum column: `'email_verify'` (existing) or `'student_activation'` (new).
- Increase `tokenHash` from 60 → 255 characters to support long activation tokens.
- Change `onDelete` from `'restrict'` → `'cascade'` so tokens are cleaned up when a user is deleted.
- Add index on `purpose` column for efficient filtering.

**Service isolation by `purpose`:**
- `EmailVerificationService` continues to work with `purpose: 'email_verify'` (6-digit codes, 10-min TTL, cooldown).
- `StudentActivationService` (new) works with `purpose: 'student_activation'` (long tokens, 1200-day expiry, one-time use).
- Both services share the same repository but are logically isolated by `purpose`.

**Migration backfill:**
- All existing `email_verifications` rows get `purpose: 'email_verify'` via migration.

## 12. Role Security Policy: Admin vs Institution_Admin

**Critical Security Constraint:** Only users with `role === 'admin'` may access admin-permissioned endpoints and the web-admin portal. Users with `role === 'institution_admin'` must NOT receive admin permissions under any circumstances.

### 12.1 Current State Analysis

The existing `AdminController` already has a class-level `@Roles('admin')` guard (`admin.controller.ts:18`). This correctly restricts ALL `/admin/*` endpoints to the `admin` role only. **This is already correct and must not change.**

However, the plan previously contained references to `role IN ('admin','institution_admin')` for web-admin portal access — **this is wrong** and must be corrected.

### 12.2 Enforced Role Boundaries

| Role | Can access `/admin/*` endpoints? | Can access web-admin portal? | Can bulk-import students? | Can approve instructors? | Can manage institutions? |
|---|---|---|---|---|---|
| `admin` | YES | YES | YES | YES | YES |
| `institution_admin` | NO | NO | NO | NO | TBD (own institution only) |
| `student` | NO | NO | NO | NO | NO |
| `instructor` | NO | NO | NO | NO | NO |

### 12.3 Implementation Guardrails

1. **Portal access enforcement (`assertPortalAccess` helper in `auth.service.ts`):**
   - `portal: 'admin'` → requires `role === 'admin'` ONLY. `institution_admin` is explicitly rejected with `WRONG_PORTAL`.
   - Do NOT add `institution_admin` to any admin-portal allowlist.

2. **Admin controller guard (`admin.controller.ts`):**
   - The existing `@Roles('admin')` class-level decorator is correct. **Do not add `institution_admin` to any admin endpoint guard.**
   - All new endpoints added to `AdminController` (CSV import, institution CRUD) inherit this guard — they remain admin-only.

3. **Institution_admin future scope (out of scope for this plan):**
   - `institution_admin` users need their own set of endpoints under a separate prefix (e.g., `/institution/*`) to manage their own institution's students, view reports, etc.
   - A dedicated `InstitutionController` with `@Roles('institution_admin')` should be created in a follow-up effort.
   - `institution_admin` users should NOT be able to access `/admin/*` endpoints, even for their own institution.
   - web-admin portal must reject `institution_admin` login attempts with a clear error: "This portal is for platform administrators only. Please contact your organization admin."

4. **Test coverage for role boundary:**
   - E2E test: `institution_admin` user calling `GET /admin/users` → 403.
   - E2E test: `institution_admin` user calling `POST /admin/institutions/:id/students/import` → 403.
   - E2E test: `institution_admin` user logging in with `portal: 'admin'` → 403 `WRONG_PORTAL`.
   - E2E test: `admin` user calling all `/admin/*` endpoints → 200.

### 12.4 Updated Portal-vs-Role Matrix

| Portal | Allowed Roles | Rejected Roles | Error on Mismatch |
|---|---|---|---|
| web-student | `student` | `admin`, `institution_admin`, `instructor` | `WRONG_PORTAL` |
| web-instructor | `instructor` (approved only) | `admin`, `institution_admin`, `student` | `WRONG_PORTAL` or `ACCOUNT_PENDING_APPROVAL` |
| web-admin | `admin` ONLY | `institution_admin`, `student`, `instructor` | `WRONG_PORTAL` |

Note: `institution_admin` does not have a dedicated portal yet — this role exists in the schema but has no frontend. Login attempts by `institution_admin` users should return `WRONG_PORTAL` for all current portals with a message: "Your account type does not have portal access yet. Contact your organization administrator."
